## Предметная область: Сбор цен конкурентов (HTTP-парсинг интернет-магазинов)

Реализован ETL/ELT конвейер для сбора и анализа данных с книжного интернет-магазина books.toscrape.com.

### Архитектура конвейера
Go-сборщик → JSON Lines → Polars (очистка) → Parquet → DuckDB/SQL → Matplotlib

### Компоненты системы

| Компонент | Технология | Назначение |
|---|---|---|
| Сборщик | Go | Параллельный HTTP-парсинг, буферизация, graceful shutdown |
| Хранилище | JSON Lines | Промежуточное хранение сырых данных |
| Очистка | Python Polars | Удаление дубликатов, преобразование типов |
| Аналитика | DuckDB | SQL-запросы к Parquet |
| Визуализация | Matplotlib | Построение графиков |

---

## Реализованный функционал

### Задания 1-3 (Go-сборщик)

- Параллельный парсинг страниц через горутины
- Канал с буфером 200 для передачи продуктов
- Пакетная запись: накопление 10 записей или сброс по таймеру 3 секунды
- Graceful shutdown: обработка SIGINT/SIGTERM с сохранением данных
- Сохранение в JSON Lines (каждая запись на отдельной строке)

### Задания 4-5 (Polars — импорт и очистка)

- Загрузка JSON Lines в DataFrame
- Удаление дубликатов
- Преобразование цены: '£51.77' → 51.77 (Float64)
- Преобразование рейтинга: 'Three' → 3 (Int64)

### Задание 6 (Агрегационный анализ)

- Группировка по рейтингу
- Вычисление средней, минимальной, максимальной цены
- Подсчёт количества книг в каждой группе

### Задание 7 (Сохранение в Parquet)

- Экспорт очищенного DataFrame в столбцовый формат Parquet
- Сохранение типов данных

### Задание 8 (DuckDB)

- Выполнение SQL-запроса к Parquet-файлу
- Замер производительности

### Задание 9 (Визуализация)

- График 1: распределение книг по рейтингу (гистограмма)
- График 2: средняя цена по рейтингу (гистограмма)
- Сохранение в PNG с разрешением 150 dpi


## Сравнение производительности
Инструмент |	Время выполнения |	Объём данных
|---|---|---| 
Polars |	< 0.1 сек |	40 строк
DuckDB |	6.0 сек |	40 строк

DuckDB показывает более высокое время на малом объёме данных из-за накладных расходов на инициализацию SQL-движка. На больших объёмах (миллионы строк) разница нивелируется или DuckDB становится быстрее.

## Выводы
1. Go с горутинами эффективно собирает данные параллельно — 50 страниц обрабатываются за < 1 секунды

2. Буферизация снижает количество операций ввода-вывода: 400 товаров → ~40 записей в файл (10x улучшение)

3. Очистка данных обязательна для корректного анализа — без преобразования типов невозможно считать среднюю цену

4. Parquet компактнее JSON и сохраняет типы данных, что ускоряет последующие загрузки

5. DuckDB удобен для SQL-аналитики, но на малых данных проигрывает Polars по скорости

6. Визуализация помогает быстро выявить неочевидные закономерности (например, низкая цена у книг с высоким рейтингом)
